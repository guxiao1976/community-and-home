## Context

主数据模块有 4 种实体走审核流程（行政区划、住宅小区、系统配置、敏感词），目前审核留痕仅依赖业务主表上的 `reviewer_id/review_time/review_notes` 字段。新增被拒绝时执行物理删除，审核信息随之丢失。`md_audit_log` 表存在但未正式使用（中间件 TODO 未完成），且其结构偏向通用操作日志，不适合审核流程追踪。

## Goals / Non-Goals

**Goals:**
- 建立独立的提交记录表，与业务表解耦，实体删除后记录保留
- 覆盖全部 4 种主数据实体的提交/审核/撤回操作
- 前端提供提交人和审核人各自的记录查看入口

**Non-Goals:**
- 不改造现有 `md_audit_log` 表或中间件
- 不修改现有审核流程的业务逻辑（仅增加记录写入）
- 不做提交记录的统计报表或图表
- 当前无 JWT 认证，user_id 从 context 取（默认 0），不做认证改造

## Decisions

### 1. 新建 `md_submission_record` 而非复用 `md_audit_log`

**选择**: 新建专用表
**理由**: `md_audit_log` 是通用操作日志（old_value/new_value JSON），缺乏审核流程专属字段（review_result、entity_name 冗余等）。专用表结构清晰，查询方便。
**备选方案**: 扩展 `md_audit_log` 增加字段 → 会导致表结构不伦不类，且现有中间件未完成。

### 2. 冗余 entity_name 和 entity_code

**选择**: 在记录表中冗余存储实体名称和代码
**理由**: 实体可能被物理删除或修改名称，冗余确保历史记录的可读性。代价是数据可能和实体当前状态不一致，但这是"快照"语义，符合审批记录的业务需求。

### 3. Model 层使用 goctl 生成 + 手动扩展

**选择**: 先手写 SQL 建表，再用 goctl 生成 model 骨架，手动添加业务查询方法
**理由**: 与项目现有模式一致。

### 4. 记录写入时机

**选择**: 在现有 Logic 层的提交/审核/撤回方法中直接调用 submission_record 的 Insert/Update
**理由**: 最小改动，不需要中间件或事件机制。单次 API 调用的性能影响可忽略。

### 5. 按提交维度记录（一次提交一条记录）

**选择**: 每次提交生成一条记录，审核时更新该记录的审核结果
**理由**: 简单直观，一次提交对应一条审核结果。不需要关联表。

## Risks / Trade-offs

- **[冗余数据不一致]** → entity_name/code 是写入时的快照，实体后续改名不会更新记录。这是符合预期的（审核记录反映提交时的状态）。
- **[批量审核性能]** → 批量审核 N 条记录会执行 N 次记录更新。N 通常很小（页面批量选择），不构成瓶颈。
- **[无软删除]** → 提交记录表不做软删除，记录永久保留。如果数据量增长，后续可考虑按时间归档。

## Migration Plan

1. 执行建表 DDL（`md_submission_record`）
2. 部署后端代码（新增 model、修改 logic、新增 API）
3. 部署前端代码（新增 Tab）
4. 无需数据迁移（新表从空开始）
