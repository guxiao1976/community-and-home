## Context

基层组织页面 (`views/grassroots/Index.vue`) 当前将省-市-区县级联选择器和提交状态筛选器放在同一视图中，查询编辑和提交两个工作流共享一个树形表格。后端 `getAdministrativeDivisions` API 已支持 `parent_id`、`level`、`submission_status` 过滤参数，`submitDivision` 和 `batchSubmitDivisions` API 已存在。

## Goals / Non-Goals

**Goals:**

- 将基层组织页面拆分为"查询编辑"和"提交管理"两个 Tab，各自独立的筛选条件和表格
- 查询编辑 Tab：以区县定位为核心，树形展示街道→社区，支持编辑/添加下级/删除，状态筛选作为辅助
- 提交管理 Tab：以状态筛选为核心，平铺展示所有匹配记录（含完整路径），支持单条提交和批量提交

**Non-Goals:**

- 不修改后端 API，复用现有接口（`getAdministrativeDivisions` 已支持所需过滤参数）
- 不修改审核中心的工作流程
- 不修改行政区划页面（level 1-3）

## Decisions

### 1. 使用 el-tabs 组件拆分视图

**选择**: 在 `views/grassroots/Index.vue` 内使用 `el-tabs`，两个 Tab 各自包含独立的筛选栏和表格。

**理由**: 用户无需跳转路由，两个工作流在同一页面内切换，操作效率高。Tab 切换不刷新页面，状态保留。

**替代方案**: 拆为两个独立路由页面 — 放弃，两个工作流紧密关联，拆路由增加导航复杂度。

### 2. 提交管理 Tab 使用平铺表格 + 完整路径列

**选择**: 提交管理 Tab 不使用树形结构，改为平铺表格，增加"完整路径"列显示 `省 > 市 > 区 > 街道 > 社区`。

**理由**: 提交时数据来自多个区县，树形展开不适用。完整路径让用户一眼识别数据来源，方便批量勾选。

**实现**: 后端已返回 `path` 字段（格式为 `/1/23/456/`，存储的是 ID）。前端需要在展示时将 ID 路径解析为名称。方案：在查询提交数据时，同时请求相关区划名称做映射；或者后端新增/扩展接口返回格式化后的路径名称。考虑到改动最小化，前端可通过已有的 `getAdministrativeDivisions({ level: 1,2,3 })` 缓存一份区划名称映射表，在渲染时将 path 中的 ID 替换为名称。

### 3. 提交管理 Tab 的数据范围

**选择**: 查询 level 4（街道）和 level 5（社区）的数据，通过 `submission_status` 过滤。默认加载"待提交"状态。

**理由**: 基层组织只涉及 level 4-5，排除 level 1-3 减少数据量。默认展示待提交数据符合"提交管理"的主要使用场景。

### 4. 批量提交复用现有 batchSubmitDivisions API

**选择**: 前端勾选多条记录后调用 `batchSubmitDivisions({ ids })`。

**理由**: 该 API 已存在且不区分 entity_type，直接传入 division ID 即可。

## Risks / Trade-offs

- **[路径 ID→名称解析延迟]** → 缓存区划名称映射表，只在首次进入提交管理 Tab 时加载一次。如果区划名称有变更，可通过刷新页面更新缓存。
- **[提交管理数据量大]** → 基层组织数据量通常可控（单个区县下街道数十条），分页加载即可。后端已支持 `page_size` 和 `page` 参数。
- **[Tab 状态不丢失]** → el-tabs 默认不销毁非活跃面板，切换 Tab 后之前的数据和筛选条件保留，用户体验好。
