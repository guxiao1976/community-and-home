# Phase 6 集成测试计划 - 用户管理

## 测试环境
- 前端: http://localhost:3001
- Identity API: http://localhost:8888
- 数据库: MySQL (identity_db)

## 测试场景

### 场景1: 获取用户列表
**目标**: 验证用户列表查询和分页功能

**步骤**:
1. GET /api/identity/users?page=1&page_size=10
2. 验证返回数据结构
3. 检查分页信息

**预期结果**:
- 返回用户列表数组
- 包含total, page, page_size字段
- 手机号正确显示

### 场景2: 创建新用户
**目标**: 验证用户创建功能

**测试数据**:
```json
{
  "phone": "13800138001",
  "nickname": "测试用户1",
  "user_type": 1,
  "password": "Test123456"
}
```

**预期结果**:
- 返回201状态码
- 返回新用户ID
- 默认is_active=true
- 默认verification_status=0

### 场景3: 更新用户信息
**目标**: 验证用户信息更新功能

**步骤**:
1. 使用场景2创建的用户ID
2. PUT /api/identity/users/{id}
3. 更新nickname和user_type

**测试数据**:
```json
{
  "nickname": "测试用户1-已更新",
  "user_type": 2
}
```

**预期结果**:
- 返回200状态码
- 用户信息已更新
- 其他字段保持不变

### 场景4: 禁用用户
**目标**: 验证用户状态切换功能

**步骤**:
1. PUT /api/identity/users/{id}/disable
2. 查询用户详情验证状态

**预期结果**:
- 返回200状态码
- is_active变为false
- 用户无法登录（如果实现了登录验证）

### 场景5: 启用用户
**目标**: 验证用户重新启用功能

**步骤**:
1. PUT /api/identity/users/{id}/enable
2. 查询用户详情验证状态

**预期结果**:
- 返回200状态码
- is_active变为true

### 场景6: 用户列表过滤
**目标**: 验证过滤功能

**测试用例**:
1. 按手机号过滤: ?phone=138
2. 按昵称过滤: ?nickname=测试
3. 按用户类型过滤: ?user_type=1
4. 按状态过滤: ?is_active=true
5. 组合过滤: ?user_type=1&is_active=true

**预期结果**:
- 每个过滤条件返回正确的结果集
- 组合过滤正确应用所有条件

### 场景7: 获取用户详情
**目标**: 验证用户详情查询

**步骤**:
1. GET /api/identity/users/{id}
2. 验证返回完整用户信息

**预期结果**:
- 返回用户完整信息
- 包含所有字段
- 不包含密码字段

### 场景8: 删除用户
**目标**: 验证用户删除功能（软删除）

**步骤**:
1. DELETE /api/identity/users/{id}
2. 尝试查询已删除用户
3. 验证列表中不再显示

**预期结果**:
- 返回200状态码
- 用户标记为已删除
- 列表查询不返回已删除用户

### 场景9: 分配角色
**目标**: 验证角色分配功能

**测试数据**:
```json
{
  "role_ids": [1, 2]
}
```

**步骤**:
1. POST /api/identity/users/{id}/roles
2. 查询用户角色验证

**预期结果**:
- 返回200状态码
- 用户角色已更新

## 测试数据准备
- 确保数据库中有测试用户数据
- 准备不同类型的用户（业主、物业、管理员）
- 准备不同状态的用户（启用、禁用）

## 验证检查点
1. API响应状态码正确
2. 返回数据结构符合预期
3. 数据库记录正确更新
4. 错误处理正确（如重复手机号）
5. 权限控制正确（如果实现）
6. 手机号脱敏显示正确
7. 状态切换UI反馈正确
