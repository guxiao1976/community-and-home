## Context

当前内容审核微服务已实现多层级审核能力（传统技术、小模型、大模型），但缺乏直观的测试和监控工具。开发和运维人员需要通过命令行或API工具手动测试，无法快速观察各审核环节的返回结果，不利于能力评估和问题诊断。

现有后台管理系统基于 Vue 3 + Element Plus + TypeScript，使用 Axios 进行HTTP请求，Pinia管理状态，Vue Router处理路由。需要在此基础上新增内容审核测试功能模块。

## Goals / Non-Goals

**Goals:**
- 提供可视化的内容审核测试界面，支持文本和图片两种内容类型
- 展示多层级审核结果（传统技术、小模型、大模型），包含各环节的原始返回数据
- 便于开发和运维人员快速测试、监控和优化审核服务能力
- 为未来大模型部署预留接口和展示区域

**Non-Goals:**
- 不实现批量测试功能（单次仅测试一条内容）
- 不提供测试历史记录存储（仅展示当前测试结果）
- 不实现审核规则配置功能（仅调用现有审核服务）
- 不支持视频、音频等其他内容类型的审核测试

## Decisions

### 1. 前端架构：单页面多Tab设计

**决策**: 使用 Element Plus 的 Tabs 组件实现文本测试和图片测试的切换，而不是分成两个独立页面。

**理由**:
- 两种测试功能高度相关，用户可能需要频繁切换对比
- 减少路由配置和页面加载开销
- 统一的结果展示组件可以复用

**备选方案**: 分成两个独立路由页面 → 增加导航复杂度，不利于快速切换

### 2. API集成：直接调用内容审核微服务

**决策**: 前端通过 Axios 直接调用内容审核微服务的HTTP接口，不经过后台管理系统的后端代理。

**理由**:
- 测试工具需要获取审核服务的原始返回数据，中间代理可能丢失细节
- 减少一层转发，降低延迟和复杂度
- 审核服务已有JWT认证，前端可直接携带token访问

**备选方案**: 通过后台管理系统后端代理 → 增加维护成本，可能丢失原始响应细节

### 3. 结果展示：分层可折叠 + JSON原始数据

**决策**: 使用 Element Plus 的 Collapse 组件分层展示各环节结果，每层提供"查看原始数据"按钮展开JSON格式的完整响应。

**理由**:
- 分层展示便于快速定位问题环节
- 可折叠设计避免信息过载
- JSON原始数据满足技术人员深度分析需求

**备选方案**: 仅展示结构化数据 → 无法满足调试需求；仅展示JSON → 可读性差

### 4. 图片上传：Base64编码传输

**决策**: 图片上传后转换为Base64编码，通过JSON格式传输给审核服务。

**理由**:
- 与文本审核接口保持一致的请求格式（application/json）
- 简化前端实现，无需处理multipart/form-data
- 审核服务已支持Base64格式图片

**备选方案**: 使用multipart/form-data → 需要特殊处理，增加复杂度

### 5. 大模型预留：前端UI预留 + 后端接口适配

**决策**: 前端UI预留大模型结果展示区域，显示"大模型未部署"提示；后端接口返回结构中包含 `largeModel` 字段（当前为null）。

**理由**:
- 避免大模型部署后需要大规模前端改动
- 用户提前了解完整的审核能力规划
- 后端接口结构稳定，便于未来扩展

**备选方案**: 大模型部署后再添加UI → 需要重新设计界面布局，影响用户体验连续性

## Risks / Trade-offs

### 风险1: 审核服务响应时间过长导致前端超时
**缓解措施**: 
- 设置合理的超时时间（文本30秒，图片60秒）
- 显示加载状态和进度提示
- 提供"取消请求"功能

### 风险2: 大图片上传Base64编码后体积过大
**缓解措施**:
- 前端限制上传图片大小（建议≤5MB）
- 提示用户压缩图片后上传
- 未来可考虑改用OSS直传方案

### 风险3: 多层级结果展示复杂，用户理解成本高
**缓解措施**:
- 提供清晰的层级标签和说明文字
- 使用颜色和图标区分不同审核结果（通过/不通过/异常）
- 添加帮助文档链接

### Trade-off: 直接调用审核服务 vs 安全性
- **取舍**: 为了获取原始数据，前端直接调用审核服务，但这要求审核服务有完善的认证和权限控制
- **当前状态**: 审核服务已有JWT认证，仅管理员角色可访问测试接口
- **未来优化**: 如需更严格的安全控制，可考虑添加IP白名单或专用测试token

## Migration Plan

### 部署步骤
1. 确认内容审核微服务已部署并可访问（小模型已部署）
2. 配置前端环境变量，设置审核服务的API端点地址
3. 部署前端代码，新增路由和菜单配置
4. 验证测试功能可用性（文本测试、图片测试）
5. 通知开发和运维团队新功能上线

### 回滚策略
- 前端代码回滚：删除新增的路由和菜单配置，重新部署
- 无数据库变更，无需数据迁移
- 审核服务不受影响，可独立回滚

## Open Questions

1. **审核服务API端点**: 当前审核服务的base URL是什么？是否需要配置不同环境（开发/测试/生产）？
2. **图片格式限制**: 除了JPG、PNG、GIF，是否需要支持WebP、BMP等其他格式？
3. **结果缓存**: 是否需要在前端缓存测试结果，避免重复测试相同内容？
4. **权限控制**: 是否所有管理员都可以访问测试功能，还是需要特定权限？
