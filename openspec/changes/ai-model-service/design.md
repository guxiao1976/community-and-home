## Context

社区平台正在引入 AI 能力，首个应用场景是敏感词自动分类（Masterdata 服务）。为避免各服务重复实现 AI 调用逻辑，决定创建统一的 AI-Model 微服务。

当前状态：
- Masterdata 服务已有敏感词 CRUD 管理，需要 AI 分类功能
- 项目采用微服务架构，服务间通过 gRPC 通信
- 已有 Identity、Masterdata、Moderation 三个微服务
- 使用 go-zero 框架，MySQL + Redis + etcd 技术栈

## Goals / Non-Goals

**Goals:**
- 提供统一的 AI 模型调用 RPC 接口，供其他微服务使用
- 支持多种 AI 模型（Claude、OpenAI、Ollama）
- 实现模型配置管理、健康检查、调用日志、成本统计
- 提供 Web 管理界面（模型配置、监控、统计）
- 实现批量调用优化（减少 API 调用次数和成本）
- 提供提示词模板管理功能
- 实现成本预警机制

**Non-Goals:**
- 不实现模型训练和微调功能
- 不实现向量数据库和 RAG 功能（后续独立服务）
- 不实现流式响应（SSE）功能（Phase 1 仅支持同步调用）
- 不实现复杂的提示词工程工具（仅基础模板管理）

## Decisions

### D1: 独立数据库 vs 共享数据库

**选择**: 独立数据库 `ai_model_db`
**理由**: 
- 遵守微服务数据独立原则，避免服务间数据库耦合
- AI 调用日志数据量大，独立数据库便于扩展和优化
- 模型配置和 API Key 属于敏感数据，独立管理更安全
**替代方案**: 使用 masterdata_db（耦合度高，不推荐）

### D2: 模型适配器架构

**选择**: 统一的 `ModelAdapter` 接口 + 多个具体实现（ClaudeAdapter、OpenAIAdapter、OllamaAdapter）
**理由**:
- 封装不同厂商 API 的差异（请求格式、认证方式、错误处理）
- 便于新增模型支持，只需实现新的 Adapter
- 统一的接口便于测试和 Mock
**接口定义**:
```go
type ModelAdapter interface {
    Call(ctx context.Context, req *CallRequest) (*CallResponse, error)
    HealthCheck(ctx context.Context) error
    EstimateCost(req *CallRequest) (float64, error)
}
```

### D3: 批量调用策略

**选择**: 应用层批量合并 + 单次 API 调用
**理由**:
- 敏感词分类场景：100 个词合并到一个 Prompt，一次 API 调用返回 JSON 数组
- 相比 100 次独立调用，成本降低 90%+，响应时间从分钟级降到秒级
- 需要在 Prompt 中明确要求返回 JSON 格式，并做结果验证
**约束**:
- 批次大小可配置（默认 100），避免单次请求过大
- 需要处理部分失败场景（某些词分类失败）
- 返回结果需要与输入词一一对应

### D4: 成本计算方式

**选择**: 基于 Token 数量 + 模型单价计算
**理由**:
- Claude/OpenAI 按 Token 计费，需要准确统计输入和输出 Token
- 不同模型价格不同，需要在模型配置中维护单价
- 成本数据用于统计和预警
**实现**:
- 调用前估算成本（基于输入 Token）
- 调用后记录实际成本（基于 API 返回的 Token 数）
- 每日汇总统计，超过阈值触发预警

### D5: API Key 管理策略

**选择**: 数据库存储 + 轮换机制
**理由**:
- 支持多个 API Key，实现负载均衡和容错
- API Key 加密存储（AES-256）
- 自动检测失效 Key（连续失败 N 次后禁用）
- 支持优先级和配额限制
**安全措施**:
- API Key 加密存储，不在日志中输出
- 管理界面仅显示 Key 的前 8 位和后 4 位
- 支持定期轮换提醒

### D6: 健康检查机制

**选择**: 定时主动探测 + 被动调用监控
**理由**:
- 定时探测（每 5 分钟）：发送简单请求检测模型可用性
- 被动监控：统计最近 N 次调用的成功率和响应时间
- 健康状态分级：Healthy（成功率 > 95%）、Degraded（80-95%）、Unhealthy（< 80%）
**告警**:
- 状态变为 Unhealthy 时触发告警
- 连续失败 3 次立即告警

### D7: 提示词模板设计

**选择**: 简单的变量替换 + 版本管理
**理由**:
- Phase 1 仅支持 `{{variable}}` 格式的变量替换
- 模板支持版本管理，可以 A/B 测试不同版本
- 模板分类（敏感词分类、内容审核、问答等）
**示例**:
```
请对以下敏感词进行分类，分类包括：政治、色情、暴力、广告、其他。
返回 JSON 数组格式：[{"word": "词1", "category": "分类1"}, ...]

敏感词列表：
{{words}}
```

### D8: 服务端口分配

**选择**: 
- ai-model-rpc: 8084
- ai-model-api: 8891
**理由**: 遵循项目端口分配规则（RPC 808x，API 889x）

## Risks / Trade-offs

**[外部 API 依赖]** → AI-Model 服务依赖外部 AI API（Claude、OpenAI），网络故障或 API 限流会影响服务可用性。缓解：支持多模型配置，自动切换；实现重试和降级机制。

**[成本控制]** → AI API 调用成本较高，滥用可能导致费用失控。缓解：实现成本预警、每日配额限制、调用审批机制（敏感操作需要审批）。

**[批量调用失败处理]** → 批量调用时，如果部分词分类失败，需要决定是全部重试还是仅重试失败的词。缓解：记录失败的词，提供手动重试接口；支持部分成功的结果保存。

**[API Key 泄露风险]** → API Key 存储在数据库中，如果数据库被攻击可能泄露。缓解：API Key 加密存储；数据库访问权限严格控制；定期轮换 Key。

**[模型响应时间不确定]** → AI API 响应时间波动较大（1-30 秒），可能导致调用超时。缓解：设置合理的超时时间（默认 60 秒）；对于批量任务使用异步处理。

**[提示词注入攻击]** → 用户输入的内容可能包含恶意提示词，干扰模型输出。缓解：对用户输入进行转义和清洗；在系统提示词中明确指示模型忽略用户输入中的指令。

## Implementation Phases

### Phase 1: MVP（最小可用产品）
- 基础服务框架（RPC + HTTP API）
- 模型配置管理（CRUD）
- Claude Adapter 实现
- 单次调用接口
- 基础调用日志
- 简单的健康检查

### Phase 2: 生产就绪
- 批量调用优化
- 成本统计和预警
- API Key 轮换机制
- 完善的健康检查和监控
- OpenAI Adapter 实现
- 提示词模板管理

### Phase 3: 优化增强
- Ollama Adapter（本地模型）
- 异步任务队列
- 流式响应支持
- A/B 测试功能
- 更详细的统计分析
- 管理界面优化
