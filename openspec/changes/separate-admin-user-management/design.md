## Context

当前系统在 `web/pc` 前端项目中有一个统一的用户管理页面，显示所有用户（包括管理员和普通用户）。identity服务提供用户查询和修改的API接口。由于管理员账户的重要性和敏感性，需要将其与普通用户分离管理，避免普通运维人员误操作管理员账户。

现有技术栈：
- 前端：Vue 3 + TypeScript + Element Plus + Vue Router
- 后端：Go + go-zero，identity服务提供用户管理API
- 权限：基于菜单权限码的访问控制

## Goals / Non-Goals

**Goals:**
- 创建两个独立的前端页面，分别管理管理员和普通用户
- 在API层面支持按用户角色类型过滤查询
- 配置独立的菜单项和权限码，实现访问控制
- 保持现有用户数据结构不变，仅在展示和查询层面做分离

**Non-Goals:**
- 不修改用户表结构或添加新的角色类型
- 不改变用户的创建、删除流程（仅关注查询和修改）
- 不涉及后端RPC服务的修改（仅API层）

## Decisions

### 1. 前端页面拆分方案
**决策**: 创建两个独立的Vue组件页面，而不是在同一页面中通过tab切换。

**理由**: 
- 独立页面可以配置独立的路由和权限码，更清晰地控制访问权限
- 避免在同一页面中混合两种用户类型的逻辑，降低代码复杂度
- 符合用户的心智模型：管理员管理和普通用户管理是两个不同的功能模块

**替代方案**: 使用tab切换在同一页面中展示两种用户，但这样无法实现细粒度的权限控制。

### 2. 用户角色类型识别
**决策**: 使用现有用户表中的角色字段（role或role_id）来区分管理员和普通用户。

**理由**:
- 避免修改数据库结构
- 利用现有的角色体系，不引入新的概念
- 需要确认identity服务中管理员角色的具体标识（如role_id=1表示管理员）

**替代方案**: 添加新的用户类型字段，但这会增加数据迁移成本。

### 3. API过滤参数
**决策**: 在用户查询API中增加可选的 `role_type` 或 `role_id` 过滤参数。

**理由**:
- 最小化API改动，保持向后兼容
- 前端可以通过传递不同的参数值来获取不同类型的用户列表
- 后端只需在SQL查询中添加WHERE条件

**实现细节**:
```go
// API定义示例
type UserListReq struct {
    Page     int64  `form:"page,default=1"`
    PageSize int64  `form:"page_size,default=10"`
    RoleType string `form:"role_type,optional"` // "admin" 或 "regular"
}
```

### 4. 菜单和权限配置
**决策**: 在菜单表中添加两个新的菜单项，分别对应管理员管理和普通用户管理，配置不同的权限码。

**权限码建议**:
- 管理员管理: `identity:admin-user:list`, `identity:admin-user:update`
- 普通用户管理: `identity:regular-user:list`, `identity:regular-user:update`

## Risks / Trade-offs

**风险1: 角色定义不明确** → 需要在实现前确认identity服务中管理员角色的具体定义（role_id或role字段的值）

**风险2: 现有用户管理页面的兼容性** → 如果有其他地方引用了旧的用户管理页面，需要评估是否需要保留或重定向

**权衡1: 功能重复** → 两个页面的代码会有一定重复（列表、编辑逻辑），但为了清晰的权限控制和用户体验，这是可接受的

**权衡2: API参数扩展** → 增加过滤参数会让API稍微复杂，但保持了向后兼容性
